# 최우선 과제 선정 기준 (Priority Ranking Spec)

> 엔진이 사용자의 할 일 중 **"지금 할 1순위"를 고르는 정확한 기준**을 명문화한 문서.
> 코드 기준: `src/main/java/com/teamsigma/taskmanager/priority/`. 변경 시 이 문서도 함께 갱신.

## 0. 핵심 원칙

"최우선 과제 = 정렬된 리스트의 맨 위(`.get(0)`)". 별도 선정 함수는 없고, 아래 **정렬 파이프라인**의 결과가 곧 우선순위다.

## 1. 정렬 파이프라인 (위 단계가 아래를 덮어씀)

`PriorityService.getPrioritizedTasks()` 순서:

| 우선 | 단계 | 규칙 |
|---|---|---|
| 1 | **핀(pin)** | 사용자가 고정한 작업은 점수·긴급도 무시하고 무조건 최상단 (수동 override) |
| 2 | **탐색(exploration)** | 5% 확률로 "최근 가장 안 한 카테고리" 1건을 **2순위로 승격**(점수 위조 없음). **1순위는 절대 안 건드림**, RED 있으면 건너뜀 |
| 3 | **RED 하드가드** | 마감 임박(RED) 작업은 비-RED 작업 **위로 절대 안 묻힘** (점수가 낮아도) |
| 4 | **점수** | 같은 밴드 안에서는 점수 내림차순 |
| 5 | **동점 처리** | 점수가 같으면 더 임박 → 더 중요 → 더 빨리 끝남(퀵윈) → `taskId` 순(결정론) |

## 2. 점수 공식 (`DefaultDynamicPriorityStrategy`)

```
score = max(0,  W1·중요도  +  W2·긴급도  −  W3·지연패널티)
```
세 요소를 모두 0~1로 정규화 후 가중. 기본 가중치 **W1=0.5 / W2=0.3 / W3=0.2** (사용자별로 학습되어 변동).

| 요소 | 계산식 | 비고 |
|---|---|---|
| 중요도 | `importance / 5` | 별점 5 = 1.0 |
| 긴급도(마감 O) | `120 / (남은분 + 120)` | 마감 시점/경과=1.0, 120분 전=0.5 |
| 긴급도(마감 X) | `delayCount / (delayCount + 5)` | 방치(aging) 기반 |
| 지연패널티 | `delayCount / 5` (상한 1.0) | **마감 있는 작업에만 적용.** 마감 없는 작업은 0 (aging과 이중 계상 방지) |

색 등급(`UrgencyEvaluator`): factor ≥ 0.66 → **RED**, ≥ 0.33 → YELLOW, 그 외 GREEN. (마감 없는 작업은 STALE/NONE)

## 3. RED 하드가드를 "점수"가 아닌 "정렬 계층"에 둔 이유

긴급도를 점수 가중치로만 처리하면, 야간 가중치 학습(`AdaptiveWeightEngine`)이 W1(중요도)을 최대 0.9까지 끌어올리면서 RED가 다시 묻힌다. 정렬 계층의 하드가드는 가중치와 무관하게 "RED는 위" 불변식을 보장한다.
→ 수용 기준: **RED 작업이 저중요·먼미래 작업 아래로 절대 내려가지 않는다.** (테스트: `PriorityServiceTest`)

## 4. 해소된 결정들 (확정 · 코드+테스트 반영됨)

| # | 결정 | 방향 |
|---|---|---|
| 중요도 vs 긴급도 | 정렬 계층 **RED 하드가드** | RED는 비-RED 위 보장, 밴드 내부는 점수(중요도 우위) 유지 |
| 동점 처리 | 결정론 tie-break | 더 임박 → 더 중요 → 퀵윈(소요시간↓) → taskId |
| `delayCount` 이중 계상 | 마감 없는 작업은 패널티 0 | aging으로 묵은 일 표면화(UI 일치) |
| 탐색 비결정성 | 1순위 보호 | 탐색 픽은 점수 위조 없이 **2순위 승격**만 |
| 미사용 신호 | `estimatedMinutes`/`requiredEnergy` 응답 노출 + 퀵윈 동점기준 | 에너지는 표시만 |
| 학습 단방향성 | **양방향 자기보정(decay)** | 정상일+W1>0.5면 기준선 0.5로 한 스텝 회귀 |

## 5. 남은 결정 (다음 차)

- **마감 곡선 절반점 120분 / 연체 가속**: 현재는 모든 연체가 긴급도 1.0(평탄)이고 RED 밴드+중요도 정렬로 처리됨 → **현 단계에선 변경 안 함**(과설계 회피). 연체 깊이 구분이 필요해지면 RED 밴드 내 "더 연체된 것 먼저" tie-break 추가.
- **에너지 매칭**: 사용자 *현재 에너지* 입력 API가 없어 보류. 입력이 생기면 피크 시간대 딥워크 가중.

## 6. 가중치 거버넌스 (조회·수동 리셋) — 구현됨

자동 학습/자기보정과 별개로, 유저가 직접 가중치를 확인·초기화하는 override 경로.

| 메서드 | 엔드포인트 | 동작 |
|---|---|---|
| GET | `/api/users/{userId}/weights` | 현재 가중치(w1/w2/w3) 조회 |
| POST | `/api/users/{userId}/weights/reset` | 기본값(0.5/0.3/0.2)으로 리셋. 프로필 없으면 400 |

- 기본값 단일 진실: `UserProfile.DEFAULT_W1/W2/W3` (학습 decay 기준선도 이 값을 참조).
- 서버: `WeightService` + `UserController`. 프론트: 설정 모달의 "기본값으로 초기화" 버튼(`api/users.js`).
