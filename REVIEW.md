## ① 한 줄 요약

**강한 의견을 가진 "랭커(ranker)"는 잘 만들었다. 그러나 제품이 약속한 "플래너"는 아직 비어 있고,
랭킹 점수 공식의 긴급도 처리가 우리 UI와 정면으로 모순된다.**
지능을 더 얹기 전에 — (1) 점수를 UI와 일치하게 신뢰 가능하게 고치고, (2) 플래너에 필요한
Task 라이프사이클 + '오늘 플랜'을 먼저 세우자.

## ② 지금 엔진이 실제로 하는 일 (검증된 사실)

이건 "task-manager"가 아니라 **우선순위 랭커**다.
입력 `(userId, List<Task>)` → 출력 `List<TaskResponse>`(점수 · 긴급도색 · 좀비플래그 · 탐색픽).

| 구성요소 | 동작 |
|---|---|
| **점수 공식** `DefaultDynamicPriorityStrategy` | `score = max(0, star*W1 + W2/(max(Δt분,0)+10.0) − delay*W3)` |
| **적응 가중치** `AdaptiveWeightEngine` (매일 0시 cron) | "picky 패턴"(쉬운 건 완료·중요한 건 스누즈) 감지 시 **W1만** ↑(부스트 0.20~0.30, 상한 0.9). 그 외 전부 "normal → 변화 없음" |
| **탐색 모드** `ExplorationService` | **5% 확률**로 최근 30일 최소 완료 카테고리의 task를 1.5x 부스트해 최상단으로 |
| **긴급도/좀비** `TaskResponseMapper` | RED < 60분 / YELLOW < 180분 / GREEN, `delayCount >= 5 → isZombie` |
| **콜드스타트** | `newUser`(가입 7일 내) → 기본 가중치(0.5/0.3/0.2), 탐색 비활성 |
| **저장** | `UserProfile`(가중치) + `UserActivityLog`(COMPLETED/SNOOZED 이벤트). **Task 자체는 비영속**(in-memory) |

## ③ 강점

1. **명확한 관점** — "나열"이 아니라 "다음에 뭘 할지 정렬". 일반 to-do 대비 차별점이 분명하다.
2. **개인화/적응** — 사용자별 가중치 학습. "쓸수록 맞아진다"는 리텐션 내러티브.
3. **UI 톤과 일치** — 좀비("5번이나 미뤄진 작업", 2.png), 탐색("오늘의 챌린지 / 새로운 도전 과제",
   4.png), AI Insights 등 엔진 개념과 UI 카피가 1:1로 매핑된다. 비전 정렬이 잘 돼 있음.
4. **콜드스타트 처리** — 신규 유저 day-1 빈 데이터를 고려한 점이 좋다.
5. **결정론적 테스트성** — `Clock`/`Random`을 빈으로 주입 → 재현·A/B 가능.

## ④ 리스크 (우선순위 높은 순)

### 🔴 1. 점수 공식의 긴급도가 사실상 무력 — UI와 모순된다 (최우선 버그)

`W2/(Δt+10)` 구조라, **60분 뒤 마감** task의 긴급도 기여 = `0.3 / 70 ≈ 0.0043`.
반면 **중요도** 기여 = `star * 0.5 = 최대 2.5`.
→ **긴급도가 중요도보다 ~500배 약하다.**

결과: "다음 주 마감 5점" 이 "10분 뒤 마감 1점"을 **항상** 이긴다.
그런데 UI(4.png)는 RED/Critical 점·뱃지로 긴급을 외친다.
**사용자는 RED 점이 박힌 task를 목록 7번째에서 보게 된다 → 신뢰 붕괴.**
추가로 점수가 정규화/무차원이라 숫자 자체가 사람에게 무의미하고 유저 간 비교도 불가.

### 🔴 2. "플래너" 절반이 통째로 비어 있다

생성/완료/삭제/스누즈/반복/리마인더/캘린더/타임블록/서브태스크가 **전부 없다.**
`delayCount`·활동로그를 **소비만** 하고 **생성하는 주체가 없다**(상태 전이 부재).
즉 추천(brain)만 만들고 본체(body)가 없는 상태. "to-do + 플래너"의 플래너가 미구현이다.

### 3. "적응"이 생각보다 얇다

학습 패턴이 'picky' **1종뿐**이고 나머지 유저는 영구 정적. 조정도 **W1만 단방향 증가**라
자기보정(감소)이 안 된다. 과보정되면 되돌릴 길이 없다.

### 4. "오늘 / 용량 / 에너지" 개념 부재

UI는 "오늘 12개", 주간진행 68%, Focus Intensity, "아침 에너지 피크 / 인지적 모멘텀"을
약속하지만 엔진엔 시간·용량·에너지·소요시간 모델이 없다.
특히 `estimatedTime`은 **로그에만 있고 점수에 미사용**.

### 5. 탐색이 설명·제어 불가한 동전던지기

전역 5% RNG가 1순위를 바꾼다. 이유 문자열·빈도 제한이 없어 "왜 이게 위에 있지?"라는
블랙박스가 된다. (UI의 `isExploration` 매핑은 좋으나, 근거가 없다.)

### 6. 좀비를 플래그만 하고 행동이 없다

`delayCount >= 5 → isZombie`로 표시만 한다. 보관/분해/재일정 판단(2.png의 "보관하기 / 계속 유지")이
엔진에 없어, 가장 어려운 행동설계를 클라이언트에 떠넘긴다.

### 7. AI 제품의 거버넌스 공백

가중치가 야간 cron으로 드리프트하는데 감사로그·설명출력·유저 override·핀 고정·리셋이 없다.
"내 정렬을 믿어라"가 핵심 가치인 제품엔 치명적이다.

## ⑤ 사용자 관점 기능 로드맵 (엔진 레벨, UI 매핑)

### P0 — 이미 절반 만든 것 / 깨진 것부터

- **점수 정규화·재밸런싱** — 각 요소를 0~1로 정규화 후 가중. 긴급도는 마감 임박 시 가파르게
  오르는 곡선으로(`1/(Δt+10)` 폐기). **수용기준: RED task가 저중요·먼미래 task 아래로 절대 안 묻힘.**
  *(UI 4.png Critical/색점이 리스트 순서와 일치해야 함)*
- **소요시간(effort) 반영** — 이미 로깅되는 `estimatedTime`을 랭킹에 사용 → "퀵윈" 노출/용량계획 기반 마련. *(4.png Focus Intensity)*
- **설명가능성 출력** — `TaskResponse`에 사유 문자열("45분 후 마감", "5회 연기", "DESIGN 3주째 안 함") 추가. *(AI Insights/챌린지 배너가 이미 사유 표시를 원함)*

### P1 — 빠진 "플래너" 본체 구현

- **Task 라이프사이클** — complete / snooze(→ `delayCount++` + SNOOZED 로그) / archive / restore
  상태전이를 엔진이 *생성*. *(2.png 보관, 3.png History/완료)*
- **"오늘 플랜" 생성** — 용량(시간예산/N개) + 에너지 입력 → 전체 랭킹이 아니라 **오늘 들어갈
  부분집합을 선택·배치.** 이게 진짜 "플래너". *(4.png "오늘 12개, ~부터 시작 추천")*
- **반복 task** — 일/주 단위(스탠드업, 주간 브리핑). 핵심 to-do 기능인데 전무. *(4.png Weekly Briefing)*
- **에너지/시간대 인지** — UI 헤드라인 약속("아침 에너지 피크"). 시간대·에너지 신호 입력 →
  피크 윈도우에 딥워크 task 가중.

### P2 — 지능·신뢰 심화

- **학습 아키타입 다양화 + 양방향 조정** — 마감주도형/꾸준형 등 추가, 가중치 감소 허용(가드레일·감쇠).
- **탐색을 설명·제한 가능하게** — 사유 문자열 + 빈도 캡 + 긴급일/신규유저 존중.
- **엔진 내 분석 집계** — "이번 주 12개", "화요일 피크", 주간 68%, 스트릭 등 UI가 주장하는
  통계의 백엔드. *(3.png View Insights, 4.png Weekly Progress 68%)*
- **유저 override/핀 + 가중치 리셋** — 수동 "이거 먼저"를 랭커가 존중. 신뢰·통제.
